# Спецификация проекта "Stock Analyzer & Screener"

## Executive Summary

Инструмент для алготрейдеров, предназначенный для поиска, отбора и мониторинга акций по пользовательским стратегиям (на дневных и 4-часовых таймфреймах), с возможностью массового скрининга и фонового мониторинга с уведомлениями в Telegram.

## Problem Statement

Алготрейдерам требуется быстрый, настраиваемый инструмент для ежедневного мониторинга рынка (вплоть до 1000 инструментов за один раз) и отслеживания узкого списка специфичных акций (2-3 тикера) в реальном времени. Ручной анализ занимает много времени и невозможен в таких масштабах из-за лимитов сторонних API (FMP).

## Success Criteria

- Способность системы сканировать 1000 акций в фоне без падения сервера.
- Корректное кэширование котировок в Redis (минимизация внешних запросов: 250 свечей для расчета, 500 для визуализации).
- Своевременные уведомления в Telegram о срабатывании стратегий по выбранным 2-3 акциям (фоновый мониторинг).
- Быстрый доступ к дашборду с готовыми сигналами сразу после входа пользователя.

## User Personas

**Алготрейдер / Разработчик стратегий (Администратор):** Технически грамотный пользователь, создающий собственные торговые алгоритмы и анализирующий результаты визуально через графики.

## User Journey

1. **Вход:** Пользователь входит в систему (с проверкой `admin_password`).
2. **Дашборд:** Сразу видит список акций, по которым сегодня прошли сигналы, и мини-графики с точками входа (срабатываниями) на индикаторах.
3. **Ручной Скан:** При необходимости переходит в раздел сканера, выбирает стратегию и запускает батч-сканирование (например, по списку S&P500). Процесс уходит в фон.
4. **Мониторинг:** В разделе мониторинга выбирает 2-3 акции для постоянного фонового слежения.
5. **Уведомления:** Получает PUSH/Telegram уведомления при совпадении критериев на 4H или 1D таймфрейме.

## Functional Requirements

### Must Have (P0)

- Дашборд со свежими сигналами и превью графиков.
- Фоновый воркер (Redis + Celery/ARQ/RQ) для массового сканирования тикеров (до 1000 штук) и фонового мониторинга.
- Умное кэширование в Redis (запрос 250 свечей на обработку, запрос 500 свечей на фронт), обновляемое по таймеру таймфрейма (каждые 4 часа или раз в день).
- Отображение исторических сигналов отслеживаемых акций прямо на свечном графике (Lightweight-Charts).

### Should Have (P1)

- Управление списками мониторинга и отложенными сканированиями из UI.
- Глобальный стейт-менеджер на фронтенде (Zustand + React Query) для переключения между графиками без лагов и морганий.
- Асинхронные HTTP запросы в Telegram бота, чтобы не блокировать потоки.

### Nice to Have (P2)

- Механика бэктестинга (векторизованного и событийно-ориентированного) для точной симуляции комиссий, проскальзываний и вывода метрик (Sharpe, MaxDrawdown).

## Technical Architecture

### Data Model

- **Пользователь (User):** email, password hash, role, telegram_chat_id.
- **Стратегия (Strategy):** JSON параметры индикаторов, код алгоритма, таймфрейм.
- **Мониторинг (Watchlist/Monitor):** символ, привязанная стратегия, статус.
- **Сигналы (SignalRecord):** символ, тип (buy/sell), время, цена, ID стратегии.

### System Components

1. **FastAPI Backend:** Принимает запросы, управляет базой данных, ставит задачи в очередь.
2. **PostgreSQL DB:** Хранение истории сигналов, параметров стратегий.
3. **Redis & Background Worker (Celery/ARQ):**
   - Выполняет очередь на сканирование 1000 тикеров.
   - Запускает крон (APScheduler или Celery Beat) каждые 4H/1D для мониторинга 2-3 акций.
   - Служит как `In-Memory Cache` для свечей (OHLCV).
4. **Next.js Frontend:** UI с `lightweight-charts`, `zustand`, `react-query`.

### Integrations

- **Financial Modeling Prep (FMP) API:** Данные (OHLCV) на 1D и 4H таймфреймах. Настроить кэширование для обхода 403 / 429 ошибок лимита.
- **Telegram Bot API:** Асинхронная отправка сообщений пользователям при срабатывании крона мониторинга.

## Non-Functional Requirements

- **Performance:** Графики должны загружаться до 1 секунды, используя закэшированные 500 свечей. Массовый скан 1000 тикеров не должен обрывать HTTP ответ фронту (должен отдавать Job ID).
- **Scalability:** Использование фоновых воркеров позволяет масштабировать парсинг независимо от API сервера.

## Open Questions for Implementation

- Будем ли мы использовать существующие фреймворки бэктестинга (типа bt, vectorbt) для проверки стратегий перед сканом, или продолжим писать собственную логику проверки (pandas_ta) для скрининга?
- Как именно пользователь формирует список из 1000 акций (текстом, CSV, или берет дефолтный S&P1000)?
