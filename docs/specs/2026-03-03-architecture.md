# Архитектурный Дизайн: Stock Scanner & Monitor

## 1. Компоненты Системы

Система разделена на 4 основных блока для обеспечения асинхронности и масштабируемости:

1. **FastAPI (API Server):** Обработка REST запросов от веб-клиента. Управление пользователями, стратегиями. Постановка задач скрининга в очередь.
2. **PostgreSQL (Database):** Персистентное хранение пользователей, параметров стратегий, списков мониторинга и исторических логов сигналов.
3. **Redis (Cache & Message Broker):**
   - Брокер очередей для фоновых задач (сканирование 1000 акций).
   - In-memory кэш котировок OHLCV (для минимизации запросов к FMP API).
4. **Background Worker (ARQ / Celery):** Асинхронные python-процессы, выполняющие тяжелый скрининг и рассылку Telegram-сигналов.

## 2. Логика Кеширования (Redis)

Для обхода лимитов FMP применяется двухуровневый кэш:

- **Кэш для расчетов (Скринер/Бэктест):** Запрашивается до 250 свечей. Ключ: `fmp:calc:{symbol}:{timeframe}`. TTL: `15 минут` для 4H, `4 часа` для 1D.
- **Кэш для визуализации (Графики):** Запрашивается до 500 свечей. Ключ: `fmp:chart:{symbol}:{timeframe}`. TTL: аналогично.

**Алгоритм запроса:**

1. Фронтенд или Воркер запрашивают данные.
2. Проверяем наличие ключа в Redis.
3. Если есть (`HIT`) -> возвращаем данные сразу (запрос к FMP не делается).
4. Если нет (`MISS`) -> делаем HTTP запрос к FMP, парсим ответ, сохраняем JSON в Redis (с TTL) и возвращаем данные.

## 3. Расчет Нагрузки и Лимитов FMP

**Вводные:**

- Лимит FMP: 750 запросов в минуту (rpm).
- Трафик: 50 ГБ в месяц.

**Сценарий массового скрининга (1000 акций):**
Если запустить скан 1000 акций без кэша и ограничения скорости, мы превысим лимит 750 rpm за 1-2 секунды.

**Решение (Rate Limiting в Воркере):**

- Фоновый воркер обрабатывает акции батчами или поддерживает жесткий rate-limit (например, используя `asyncio.Semaphore` или библиотеку `limiter`).
- Ограничение устанавливается на уровне **10-12 запросов в секунду** (600-720 rpm), оставляя запас в 30 rpm для пользовательского серфинга (открытие графиков).
- Время выполнения 1000 сканирований "на холодную" (без кэша) = `1000 / 12 = ~83 секунды (1.5 минуты)`.
- Время выполнения "на горячую" (с кэшем Redis) = `~2-3 секунды` (т.к. к FMP нет обращений, расчет индикаторов идет локально на CPU).

**Потребление Трафика (50 ГБ):**

- 1 запрос OHLCV на 500 свечей "весит" примерно 20-30 КБ (в формате JSON).
- 1 скан 1000 акций = `1000 * 30 КБ = 30 МБ`.
- Месячный лимит (50 ГБ) позволяет выполнить примерно `50,000 / 30 = 1666` "холодных" сканирований по 1000 акций в месяц (около 55 полных сканирований рынка каждый день), что более чем достаточно при использовании кэширования.

## 4. Telegram Notification (Сигнальный Бот)

- **Отдельный токен бота:** `8609615438:AAGiMnCXVv9whnVi3-FOUwEh1hVtauivaHI`.
- Воркер (Cron/Beat) каждые 4 часа проходит по списку "Monitor" (2-3 акции пользователя), запрашивает свежие свечи, прогоняет через `pandas_ta` по привязанной стратегии.
- При наличии пересечения (сигнала Buy/Sell) на последней закрытой свече, воркер асинхронно через `httpx.AsyncClient` шлет GET/POST запрос к `api.telegram.org/bot<TOKEN>/sendMessage`.
